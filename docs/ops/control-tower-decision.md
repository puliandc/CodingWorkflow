# 📡 Control Tower 专属能力归属决策记录 (ADR-003)

## 一、 背景与现状
在进行 `9zsyqss` 项目接入过程中，开发团队曾经将 `Control Tower`（3D 图形遥测监控与并发限流中枢）作为 `CodingWorkflow` 引擎的拟下沉能力进行评估。然而，在端到端功能验收中发现，`Control Tower` 的实现深度依赖 `9zsyqss` 的私有数据库 Schema、高并发限流算法以及 WebGL 视图。如果强行将其硬编码或以插件强形式内置在通用引擎中，会导致：
1.通用引擎臃肿，偏离 Epic “保持精简” 和 “零硬编码” 的底线原则。
2. 导致非 9zsyqss 项目（如 P1914 等 Godot 工程）接入时引入无意义的依赖和逻辑死锁。

## 二、 架构裁决与归属决策
经过架构审查，我们正式做出如下**归属裁决**：
- **Control Tower 等 9zsyqss 专属高阶遥测与业务监控能力，决定 100% 显式保留为 9zsyqss 本地专属 skill（位于 9zsyqss 仓库的 `.claude/skills/` 或是本地专属插件中）**。
- **通用 `CodingWorkflow` 研发纪律引擎保持纯粹、极简和 100% 跨项目通用下沉**，引擎底座仅维护核心的六大状态机生命周期、并发契约锁、只读 precheck 门禁与动态自进化 compiler 等通用红线纪律。

## 三、 结论与意义
本决策有效保护了引擎的设计边界，达成了 100% 的项目适配解耦。
- 通用引擎：**零硬编码、零项目专属业务偏好**。
- 特定项目：通过在本地扩展专属 `skills`，自主、透明地实现高阶定制功能，完美做到了“引擎极简，项目灵活”。
